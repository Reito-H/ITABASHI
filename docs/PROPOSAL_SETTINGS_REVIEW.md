# 設定ページ 見直し提案書

**作成日**: 2026-07-19
**対象**: Benten管理システム「設定」ページ（全18カード）とその周辺

> **✅ 実施済み（2026-07-19）**: ユーザー承認のもと、提案A〜Gをすべて実装しました。
> 一部は安全のため方式を調整しています（C-2は「一本化」ではなく相互リンク＋通知設定ページで全種類管理、Fのページ再読み込み廃止は報告センターのみ適用）。詳細は実装時の作業記録を参照。

---

## 1. 何を調べたか

寝ている間に、次の4つの目線で設定ページ全体をチェックしました。

1. **無駄な機能はないか**（使われていないページ・コード）
2. **まとめられる機能はないか**（似たページが別々に存在していないか）
3. **通信・UI/UXは最適か**（余計な通信、分かりにくい画面）
4. **マニュアルは最新か**（チュートリアル・README・仕様書）

調べたファイル: `admin.ts` / `admin_liff.ts` / `admin_extra.ts` / `admin_accounts.ts` / `admin_benten.ts` / `admin_kancho.ts` / `permissions.ts` / `index.ts` / `layout.ts` / `cron.ts` / 各マイグレーションSQL / docs一式

## 3. 提案の詳細

### 提案A: 権限チェックの抜け穴をふさぐ（重要度: 高）

**今の状態**
このシステムのAPI（画面の裏でデータを保存する窓口）には、新旧2種類あります。

- **新しいAPI** `/{秘密パス}/admin/api/...` → アカウントごとの権限チェック**あり**（閲覧だけの人は編集できない）
- **古いAPI** `/api/coaches`、`/api/schedule-types`、`/api/period-settings`、`/api/notifications`、`/api/instructors` など → ログインしてさえいれば**誰でも編集できる**

つまり「閲覧のみ」に制限したアカウントでも、画面のボタンではなく古いAPIの窓口を直接たたけば、シフト区分や研修担当などを書き換えられてしまいます。

**提案**
古いAPIにも、新しいAPIと同じ権限チェック（`permissions.ts` の仕組み）をかける。対象は上記5つ＋ `/api/instructor-invite` など設定系のAPIです。

---

### 提案B: マニュアル（チュートリアル）を今の機能に合わせる（重要度: 高）

**今の状態**
設定 → チュートリアルの表紙には「最終更新: 2026年7月」とありますが、中身は古いままです。

- **章がまるごとない機能**: 班長シフト / 点検管理 / マニュアルBot / LINE利用状況 / 社員絞り込み検索 / 忘れ物・事故・違反の報告 / ベンテンクラブシフト
- **「1-12 設定」の一覧表が今の設定ページと合っていない**:
  - 今は存在しない「車両検索管理者」が載っている
  - 逆に「アカウント権限管理」「LINEリフ権限管理」「忘れ物/事故/違反報告一覧」「違反種類」「ベンテンクラブシフト」「営業所」が載っていない
- 設定トップの「LINE通知設定」カードの説明も「班長向け定時通知」のみで、実際はベンテンシフト通知・班長出勤通知も含まれています

また、リポジトリのドキュメントも古いです。

- `README.md`: 旧名称「新人離職防止システム」のまま。構成も「Cloudflare Pages」と書かれていますが実際はWorkersです
- `docs/SPECIFICATION.md`: 6月4日から更新なし（その後に追加された機能が多数未記載）

**提案**
1. チュートリアルに不足している章を追加し、「1-12 設定」の表を現在の18カードに合わせて書き直す
2. README.md の名称・構成を現状に合わせる
3. SPECIFICATION.md は「歴史的資料」と割り切るか、最新化するかを決める（メモリ上の確定仕様は別管理されているので、廃止宣言だけでも可）

---

### 提案C: LINE通知設定を1か所にまとめる（重要度: 中）

**今の状態**
通知は現在4種類あります（朝の出勤レポート / 嫌なこと報告アラート / ベンテンシフト通知 / 班長出勤通知）。

ところが「設定 → LINE通知設定」ページは、最初の2つにしか日本語名を用意していません。そのため残り2つは **`benten_shift_daily`、`kancho_attendance` という英語のまま**、説明なしで表示されます。

さらに、この2つは別の画面（ベンテンクラブシフト設定・班長シフト画面）からもON/OFFできるため、**同じスイッチが2か所にある**状態です。片方で変えたことにもう片方で気づけません。

**提案**
1. まず最低限: LINE通知設定ページに日本語名と説明を追加する（「ベンテンシフト通知」「班長出勤通知」）
2. できれば: 通知のON/OFF・時刻変更は「LINE通知設定」ページに一本化し、各機能側のページからはリンクだけ置く

---

### 提案D: 報告系3ページの統合と、ダッシュボードへの表示（重要度: 中）

**今の状態**
「忘れ物報告一覧」「事故報告一覧」「違反報告一覧」の3ページは、作りがほぼ同じ（一覧表＋解決ボタン＋削除ボタン）で、コードもほぼコピペです（各約150行 × 3）。細かい差もあります。

- 忘れ物にはフィルタボタン（対応中/解決済など）があるのに、事故・違反にはない
- 3つとも「設定」の中にあるが、実際は毎日見る**運用ページ**であって設定ではない
- ダッシュボードの「未対応の報告」は嫌なこと報告だけを数えていて、**未対応の事故報告があってもトップ画面に出てこない**

**提案**
1. 3ページを1つの「報告センター」ページ（タブ切り替え: 忘れ物/事故/違反）に統合する。サイドバーの「報告一覧」（嫌なこと報告）と並べる形も良い
2. フィルタボタンを3種類とも共通で付ける
3. ダッシュボードに「対応中の忘れ物/事故/違反の件数」を追加する（事故の見落とし防止に効果大）
4. コード上も共通化でき、3ページ分の重複（管理者名の取得処理なども3回コピペ）が消える

---

### 提案E: 使われていない旧設定ページの削除（重要度: 低・効果は確実)

**今の状態**
`admin.ts` に `/settings/legacy` という旧設定ページ（シフト区分・研修担当・班長指導者・月度をまとめた古い画面、**約330行**）が残っています。どこからもリンクされておらず、知っている人がURLを直接打たない限り誰も開けません。

同じく `layout.ts` の `LOGO_B64`（約16KBの画像データ）はどこからも使われていない死にコードです。

**提案**
両方とも削除する。動作への影響はありません（リンク元ゼロを確認済み）。

---

### 提案F: 通信の無駄をへらす（重要度: 低〜中）

**今の状態と提案**

| 場所 | 今 | 提案 |
|------|----|------|
| 月度設定の保存 | 12か月分を**12回**順番に通信 | 1回の通信でまとめて保存（営業所ページは既に1回方式なので、同じ形に揃える） |
| シフト区分の一括保存 | 行の数だけ順番に通信 | 同上（1回にまとめる） |
| 各ページの保存・削除後 | 毎回ページ全体を再読み込み（`location.reload()`） | 変わった行だけ書き換える（体感速度アップ。優先度は低くてOK） |
| システムステータスのAPIチェック | 社員一覧APIをフル取得して速度測定 | 件数だけ返す軽い窓口にする（負荷減） |
| ステータスページのQRコード | 外部CDN（jsdelivr）からライブラリを毎回読み込み | 自前で配信 or サーバー側で画像生成（CDN障害時も動く） |

※ 全体としては「サーバーでHTMLを作って返す」方式で通信回数はもともと少なく、大きな問題はありません。上記は細かい改善です。

---

### 提案G: 設定トップの並びを整理する（重要度: 低〜中）

**今の状態**
設定トップは18枚のカードが縦一列に並んでいるだけで、「毎日見るもの」と「一度設定したら触らないもの」が混ざっています。

**提案**
カードを見出し付きでグループ分けする（並べ替えるだけで、機能変更なし）。例:

- **日々の運用**: 忘れ物報告 / 事故報告 / 違反報告 / お知らせ配信
- **権限・アカウント**: アカウント権限管理 / LINEリフ権限管理
- **シフト関連の設定**: シフト区分 / 研修担当 / 班長・指導者 / 月度設定 / ベンテンクラブシフト
- **LINE関連**: LINE管理 / LINE通知設定
- **マスタ**: 営業所 / 違反種類・点数
- **ガイド・システム**: チュートリアル / 車番検索ガイド / システムステータス

あわせて小さな重複の解消: 「お知らせ配信」は**サイドバーと設定カードの両方**にあります。どちらか片方（サイドバー推奨）に絞ると迷いません。

## 4. やらなくてよいと判断したこと

- **設定ページの大規模な作り直し**: 現在のカード方式自体は分かりやすく、問題ありません
- **シフト区分/研修担当/班長・指導者/月度の4ページ統合**: 昔の統合ページ（legacy）を分割した経緯があるので、今の分割のままで良いと判断
- **フレームワーク導入などの技術刷新**: 通信量・速度に大きな問題はなく、不要
