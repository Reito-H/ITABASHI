## 0. 現状把握（実装前提の整理）

- `sales_records` テーブルは**既存**。新人（`newcomer`ロール）が LINE Bot の会話フロー（`src/line_bot.ts` `handleNewcomer()`）で
  日付・金額(税込)・乗車回数・走行距離を記録済み。管理画面 `/sales`（`src/routes/admin_extra.ts` + `src/html/sales.ts`）で
  月度別グラフ・CSV出力が可能。今回はこの土台を**拡張**する。
- `line_liff_users.role` は TEXT カラム（CHECK制約なし）。過去にも `benten_member` / `benten_shift_master` を
  **スキーマ変更なしで値追加のみ**で対応した実績あり（migration_025）。今回も同じ方式で `crew_member`（乗務社員）を追加する。
- **勤務区分コードは `employees.work_schedule` と同じ `a/b/B/D/H` を再利用する**（ご確認済み仕様）：
  **小文字＝日勤、大文字＝隔日勤務**。`a`＝昼日勤、`b`＝夜日勤、`B`/`D`/`H`＝隔日勤務の各時間帯。
  大文字小文字でカウント重み（1.0 / 0.5）を判定できるため、列挙値を新設せず1文字コードのままシンプルに扱う。

## 2. DB設計（migration_026 想定）

### 2-1. 売上記録の拡張（既存 `sales_records` にカラム追加）

```sql
ALTER TABLE sales_records ADD COLUMN duty_code TEXT
  CHECK(duty_code IS NULL OR duty_code IN ('a','b','B','D','H'));
```

| duty_code値 | 意味 | カウント重み |
|---|---|---|
| `a` | 昼日勤 | 0.5 |
| `b` | 夜日勤 | 0.5 |
| `B` | 隔日勤務B | 1.0 |
| `D` | 隔日勤務D | 1.0 |
| `H` | 隔日勤務H | 1.0 |
| `NULL`（既存レコード・後方互換） | 未設定（旧新人データ） | 1.0扱い（デフォルト） |

- 判定ロジックは**大文字小文字のみ**で分岐できる（例: `duty_code = UPPER(duty_code)` なら隔日勤務＝1.0、それ以外(小文字)なら0.5）。
- 社員登録済みの `employees.work_schedule` が同じコード体系であれば、その値をデフォルト選択肢として
  クイックリプライの初期候補に使ってもよいが、**実際の乗務は日によって変わりうる**ため、都度選択可能な項目として独立させる
  （`sales_records.duty_code` は `employees.work_schedule` を都度コピーするだけで、参照制約は持たせない）。

- **税抜金額は列を持たない**（コンパクト設計）。`amount`（税込）から表示・集計時に都度計算する：
  `税抜 = ROUND(amount / 1.1)`。会計上の端数処理と厳密一致させる必要が出た場合は、後から計算方式を変更しても
  過去データの再計算だけで済む（列がないので移行不要）。
- 月度合計カウント（実働換算）は集計時に `SUM(CASE WHEN duty_code IS NULL OR duty_code = UPPER(duty_code) THEN 1.0 ELSE 0.5 END)` で算出。
- 既存 `UNIQUE(emp_id, date)` はそのまま維持（1日1乗務の前提を継続）。

### 2-2. ODOメーター記録（新規テーブル）

```sql
CREATE TABLE IF NOT EXISTS odo_records (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  emp_id      INTEGER NOT NULL REFERENCES employees(id),
  odo_start   INTEGER NOT NULL CHECK(odo_start BETWEEN 0 AND 999999),
  odo_end     INTEGER CHECK(odo_end IS NULL OR odo_end BETWEEN 0 AND 999999),
  distance_km INTEGER,                          -- odo_end - odo_start（確定時に計算して保存）
  started_at  TEXT DEFAULT (datetime('now', 'localtime')),
  ended_at    TEXT,
  created_at  TEXT DEFAULT (datetime('now', 'localtime'))
);
CREATE INDEX IF NOT EXISTS idx_odo_open ON odo_records(emp_id, odo_end);
```

**設計方針（コンパクト＋大容量対応）：**
- 1レコード＝1乗務（ODO開始〜終了の1サイクル）。カレンダー日付での紐付けはしない
  （隔日勤務は日付を跨ぐため、「今日の記録」ではなく「**未確定（`odo_end IS NULL`）の最新レコードがあるか**」で
  開始／終了を判定する。これによりテーブル設計を1テーブルのみに保てる）。
- 走行距離は都度計算・保存（読み取り時の再計算コストを避けるため、確定時に1回だけ計算）。
- INTEGER型のみで構成し6桁上限をCHECK制約で強制 → 行サイズが小さく、乗務員数×日数が増えても
  D1（SQLite）で長期間問題なくスケールする。将来的に古い年度分を別テーブルへアーカイブする運用も容易。

## 4. 機能仕様 その2: ODOメーター記録機能（2026-07-13 確定版）

### 4-1. LINE Bot 会話フロー

「ODO」リッチメニューボタン押下時の処理（`handleSalesOdoFlow()`内、状態機械は不要・都度DBを見て判定）：

1. `SELECT id, odo_start FROM odo_records WHERE emp_id = ? AND odo_end IS NULL ORDER BY id DESC LIMIT 1`
2. **該当なし（1回目の押下）** → 数値入力待ち状態（`line_conv_states` に `odo_awaiting_start` をセット）
   → 数値（6桁以内。5桁等も可）を受信 → `INSERT INTO odo_records (emp_id, odo_start)` → 「ODO始: ○○○○○ を記録しました」
3. **該当あり（2回目の押下）** → 数値入力待ち状態（`odo_awaiting_end`、対象レコードIDを保持）
   → 数値受信 → **`odo_end >= odo_start` を検証**（下回る場合は「ODO終がODO始より小さいです。入力し直してください」で再入力させる）
   → 自動返信：
     ```
     🚕 ODO記録が完了しました
     ODO始: 12345
     ODO終: 12501
     走行距離: 156km
     ```
   → **返信後、そのレコードはDELETEして削除する**（ユーザー確認済み・集計に使わないその場限りの記録のため）

**「ODO始」送信後、他機能を使っても保持される仕組み**：
「未確定(odo_end IS NULL)レコードの有無」をDBで判定する設計のため、`line_conv_states`（会話の状態）が`idle`に
戻っていても保留中のODO開始記録はDBに残り続ける。ODO入力待ち状態（`odo_awaiting_start`/`odo_awaiting_end`）の間に
**数字以外の入力**（＝他機能のボタン等）が来た場合は、入力待ちを静かに解除して`false`を返し、
そのメッセージを本来の役割別ハンドラーに渡す（他機能を誤ってブロックしない）。前回のODO終の値を次回の
ODO始に引き継ぐような継続性は持たせない（車両が変わるため不要、との確認済み）。

### 4-2. 売上サマリーとの連携（廃止）

~~走行距離を月度サマリーでODOから集計する~~ → **廃止**。ODOレコードは返信後に削除するため、月次の走行距離集計機能は
実装しない（`/liff/sales`のサマリータブから「走行距離(ODO)」カードは削除済み）。ODOはあくまで乗務員がその場で
距離を計算するための単発ツールという位置づけ。

### 4-3. 既知の制約（設計上のトレードオフとして明記）

- オドメーターの桁溢れ（999999→000000ロールオーバー）は非対応。実務上6桁で運用されているとのことなので当面問題ないが、
  もし発生した場合は管理画面から手動補正する運用とする（自動補正ロジックは複雑さの割に発生頻度が低いため見送り）。
- **強制クローズの管理画面は今回作らない（決定事項）**。「未確定レコードが残ったまま次の乗務を開始した」ケースは
  当面 `wrangler d1 execute` での手動DB修正で対応する（発生頻度を見てPhase外で画面化を検討）。

## 6. GPT画像生成プロンプト（リッチメニュー用）

日本語テキストのAI画像生成は文字化けしやすいため、**アイコンのみを生成し、ラベル文字は後から画像編集ツールで重ねる**方式を推奨します
（プロンプト内に注記あり）。

## 7. セキュリティ設計

- **なりすまし防止**：全ての新規API・Bot処理で `emp_id` はクライアント入力を一切信用せず、
  `Authorization: Bearer` トークン→ `/v2/profile` → `line_liff_users` 経由で解決した値のみを使用（既存LIFF認証方式を完全踏襲）。
- **ロールチェックはサーバー側で必須**：`crew_member` 等の許可ロール判定はUIで隠すだけでなく、
  API/Bot処理の入口で明示的に403相当の拒否を行う（ベンテンクラブ機能と同方針）。
- **入力値検証**：売上金額・ODO値はいずれも `0〜999999` の範囲チェックをサーバー側で必須実施（クライアント側チェックのみに依存しない）。
- **二重送信対策**：売上は既存の `UNIQUE(emp_id, date)` を継続。ODOは「未確定レコードが存在する場合は新規開始させない」ロジックで
  同一乗務員の多重オープンを防止。
- **PDF公開URLの保護**：ベンテンクラブPDF機能で確立済みの HMACトークン方式（`bentenPdfToken` 相当）をそのまま流用し、
  推測可能な連番URLでの第三者アクセスを防止。
- **監査性**：`odo_records` に `started_at` / `ended_at` を保持することで、後から異常値（極端な走行距離等）を
  管理画面で検知しやすくする（今回のスコープでは検知UIまでは含めないが、列設計として将来対応可能にしておく）。
